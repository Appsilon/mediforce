How we think about validated AI

You can't validate AI the way
you validate software.
So we validate the process.

AI gives you a draft, a suggestion, a classification. But in pharma, you can't use it until you can prove the process around it was sound. That's the problem we solve.

The situation right now

Your team wants to use AI. Your compliance team wants to know it's safe. There is no FDA playbook for this. No standard. No checklist.

But there doesn't need to be — because pharma has solved a version of this problem before. Every R package used in a submission is "validated." Every LIMS, every EDC, every SAS script. These tools went through qualification, testing, documentation — and then a human used them within a controlled process and took responsibility for the result.

The difference with AI is that the tool itself is non-deterministic. Same input, different output every time. So you can't just qualify the tool once and move on. You need a process that catches problems continuously — not just once during validation.

Our approach: make every AI output earn its way through

In Mediforce, an AI agent doesn't just produce an output and hand it off. The output goes through a pipeline of checks before anyone can use it. Here's what that looks like:

Agent produces a result
The AI drafts a document, classifies data, generates a recommendation — whatever the workflow step requires. Everything is logged: model version, prompt version, input data, raw output.
Automated
Guardrails check the output
Does the output have the right structure? Are values within expected ranges? Does it reference things that exist in the source data? Is there patient data that shouldn't be there? If any check fails, the output is blocked before it goes further.
Automated
The platform enforces the next step
If this step is configured as human-in-the-loop, the platform will not let it proceed without approval. No workaround, no override. The autonomy level you set is the autonomy level that happens.
Platform enforced
A qualified person reviews
They see exactly what the agent proposed, highlighted against the source data. They can approve, modify, or reject. Every change is tracked — what the AI said vs. what the human decided. The platform records who signed off, when, and what the action meant.
Human decision
The platform checks if the review was real
Did the reviewer spend enough time? Do they approve everything unchanged, every single time? The platform tracks patterns. If someone is rubber-stamping, the system flags it. Genuine oversight isn't optional — it's enforced.
Anti-rubber-stamping
The result is ready to use
Now you have an output that's been through automated checks, reviewed by a qualified person, and backed by a complete audit trail. You can use it. You can defend it. You can show an auditor exactly what happened.

This isn't just about compliance

The reason we decompose work into steps isn't just for the audit trail. There are practical reasons this is better than giving an agent a big task and hoping for the best:

Smaller blast radius

If an agent makes a mistake in step 3, you catch it there. You don't discover it after the whole process ran.

Different experts for different steps

A medical writer reviews the narrative. A statistician reviews the numbers. Each person sees only what they need to judge.

Agents don't judge their own work

The agent that produces a draft doesn't review it. A different agent — or a human — does. Like separation of duties in GMP.

Security by design

Each agent can only access the data it needs. Working on Study A? It can't see Study B. Scoped per schema, per table, per study.

How we think about reproducibility

In traditional pharma: same code, same data, same result — even 20 years from now. With AI, that's impossible. The model might not exist next year.

But what's reproducibility actually for? It's so an auditor can verify the result is legitimate. In our model, you don't need to re-run the agent. You need to reconstruct the decision:

Here's the data. Here's what the agent proposed. Here's what the reviewer changed. Here's the final version. Here's who signed off.

The audit trail is the reproducibility mechanism — not re-execution. Everything is stored: input, agent config, raw output, human modifications, final artifact. Even if the model no longer exists, you can prove what happened.

Not everything needs the same rigor

If the output feeds into an FDA submission, you want the full stack: agent qualification, automated guardrails, human review with anti-rubber-stamping. Autonomy L1–L2 only.

If it's supply chain logistics for clinical supplies, that's your internal process — no predicate rule requires you to validate it. You still get the platform, you still get the audit trail, but the rigor is proportionate to the risk.

This isn't our invention. It comes from ICH E6(R3), which introduced "critical-to-quality factors" — focus rigor where it matters for patient safety and data reliability. We just made it a configuration setting in the platform.

Where we are

This is our point of view, not an industry standard. There isn't one yet. We're building Mediforce in conversation with pharma teams, regulatory consultants, and people who've been doing GxP validation for longer than we've been writing code.

If you see gaps in this thinking — especially if you think we're wrong about something — that's the conversation we want to have.
