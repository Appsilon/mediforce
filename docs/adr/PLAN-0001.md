# PLAN-0001 — Implementation plan for ADR-0001

Companion to [`0001-firestore-to-postgres.md`](./0001-firestore-to-postgres.md).
This document is **exhaustive** — the implementing agent should follow it
step-by-step and never assume something can be skipped because "it's obvious".

Cross-reference for domain language: [`../../CONTEXT.md`](../../CONTEXT.md).

---

## 0. Dual-code, not dual-write

During the implementation phase covered by this plan (§5 per-repo PRs through
the cutover in §8), every repository PR ships **both** the existing Firestore
implementation and the new Postgres implementation. The choice between them
is a one-line ternary at the repo's declaration site in
`getPlatformServices()`, gated on `STORAGE_BACKEND`. The flag default stays
`firestore`, so in production:

- The Postgres path exists in code but is **dormant** — no production traffic
  reaches it.
- There is no dual-write. No "write to both, read from one". No
  reconciliation job. No runtime mix of two truths.
- The only Postgres data during this phase lives in CI (parity tests) and in
  developer machines (`docker-compose up postgres`).

The flag flips exactly once, during the big-bang cutover (§8). That is when
Postgres data first exists in production and the Firestore path goes cold.
From that moment on, joins, FKs, and multi-table transactions are all
available across the whole schema — no "lazy" FK plumbing, no awaiting peer
tables, no progressive enablement.

**Consequence for schema authoring (§1, §5):** every PR's Drizzle schema
should declare FKs against tables defined in earlier PRs' schemas (or
against `workspaces` if it lands first). FKs need only the **schema** peer
in Postgres, not the runtime peer — there is no runtime dual-state to
respect. This is why §5.2 leads with `workspaces`: the table every other
table FKs into.

## 1. Storage shape decisions (per aggregate)

### 1.1 General conventions

- Every table has:
  - `id uuid primary key default gen_random_uuid()`
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()` with a trigger to bump on `UPDATE`
  - Any tenant-scoped table also has `workspace text not null` (replaces today's `namespace` column; same semantic, new name).
  - Soft-delete tables also have `deleted_at timestamptz` and (where the schema currently has `archived`) `archived_at timestamptz`. Active rows have both `NULL`.
- All indexes on tenant-scoped tables lead with `workspace` (e.g. `(workspace, status, created_at desc)`).
- Soft-delete tables get a `WHERE deleted_at IS NULL AND archived_at IS NULL` partial index for hot read paths.
- Polymorphic / "schemaless" data lives in `jsonb` columns. Query-worthy fields are expanded into typed columns (the **hybrid pattern**).

### 1.3 What we explicitly do NOT change

  read whole, mutated through a single "publish new version" path.

  `${steps.X.output.Y}` reads the whole map; per-key indexing would be premature.
- `StepExecution.output` stays polymorphic (`jsonb`). Different step types yield
  different shapes.

## 2. Fail-proof repository pattern

### 2.1 Base class sketch

The base class is reshaped from the original singular-workspace sketch to
take a `CallerIdentity` value (see
[ADR-0004](./0004-scoped-data-access-authorization.md) §7), so the same
filter form maps onto both today's multi-workspace endpoints (e.g.
`GET /api/tasks?role=…` lists tasks from every workspace the caller
belongs to) and the future Postgres RLS policy
(`workspace IN (SELECT workspace FROM workspace_members WHERE user_id = current_user_id())`).
A `scopedTo(handle)` mode covers URL-driven `/{handle}/…` routes after the
middleware has confirmed membership of `handle`.

```ts
// packages/platform-infra/src/postgres/repositories/base.ts
export abstract class AuthorizedRepository<TRow extends { workspace: string }> {
  constructor(
    protected readonly db: Database,
    protected readonly table: PgTable,
    protected readonly caller: CallerIdentity,
  ) {}

  /** Default scoped read — caller-bound and excludes tombstones. */
  protected scoped() {
    const base = this.db
      .select()
      .from(this.table)
      .where(
        and(
          isNull(this.table.deletedAt),
          isNull(this.table.archivedAt),
        ),
      );
    if (this.caller.kind === 'apiKey') return base;
    return base.where(inArray(this.table.workspace, [...this.caller.workspaces]));
  }

  /** Pin to a single workspace — used by /{handle}/… routes after middleware
   * has verified the caller is a member of `handle`. */
  protected scopedTo(handle: string) {
    if (this.caller.kind !== 'apiKey' && !this.caller.workspaces.has(handle)) {
      throw new ForbiddenError();
    }
    return this.db
      .select()
      .from(this.table)
      .where(
        and(
          eq(this.table.workspace, handle),
          isNull(this.table.deletedAt),
          isNull(this.table.archivedAt),
        ),
      );
  }

  /** Audit-logged opt-out for cross-workspace public discovery. */
  protected crossWorkspacePublic(condition: SQL) {
    auditLog({ action: 'cross_workspace_public_read', condition: String(condition) });
    return this.db.select().from(this.table).where(condition);
  }

  /** Audit-logged opt-out for admin / cleanup use. */
  protected includeDeleted() { /* … */ }
  protected includeArchived() { /* … */ }
}
```

### 2.2 Rules of use

- Repositories **never** expose `this.db` outside themselves. The base class
  closes the `db` member as `protected`.
- Caller code (services, route handlers) **never** imports Drizzle's `db` and
  **never** writes SQL by hand. It calls repository methods.
- Adding a new repository method that has to ignore the default filters
  requires:
  - Naming it explicitly (`discoverPublic…`, `…IncludingDeleted`, etc.)
  - Calling one of the base-class opt-outs (which emits an `audit_event`)
- Linter rule (custom ESLint or a `grep` pre-commit hook): forbid `db.select`,
  `db.insert`, `db.update`, `db.delete` outside
  `packages/platform-infra/src/postgres/repositories/`.

### 2.3 Multi-tenancy enforcement layers

| Layer | What it does |
|---|---|
| App middleware | Resolves the calling user → `CallerIdentity` (uid + the set of member workspaces). Rejects any request whose `:handle` URL segment is not in that set. |
| Repository base class | Filters every default read/write by `workspace IN $callerWorkspaces` (unrestricted for `caller.kind === 'apiKey'`). |
| Postgres RLS (Phase 2, separate ADR) | Belt-and-suspenders: even a bug in the repo cannot leak across workspaces. |

This ADR delivers the first two; RLS is a later, additive change. The
caller-set form maps directly onto the eventual RLS policy
(`workspace IN (SELECT workspace FROM workspace_members WHERE user_id = current_user_id())`).

## 3. Realtime swap

### 3.0a Hard dependency — discovered during PR2 implementation

Migrating each `onSnapshot` hook to `useSWR(url, …)` requires a `mediforce.X.Y()`
client method backed by a platform-api handler + Zod contract + route adapter.
**Half of the consumers don't have that surface yet:** `agent-runs.list`,
`agent-runs.getByStepId`, `processes.list`, `namespaces.*`, `workflow-meta` reads,
broader `tasks.list` (cross-instance, cross-role). Adding 6-8 net-new headless
endpoints belongs in [`docs/headless-migration.md`](../headless-migration.md)
Phase 2.5 (`Definitions, agents, admin, users`), not in this storage-swap ADR.

**What PR2 ships:** the SWR foundation (`useLive` / `useRetro` helpers in
`packages/platform-ui/src/hooks/use-swr-mediforce.ts`) + migration of the one
hook that already has a complete client surface (`use-audit-events` via
`mediforce.processes.listAuditEvents`). The remaining 8 hooks and 3
page-component listeners stay on `onSnapshot` until Phase 2.5 fills the
client surface.

**Verification §10's `grep onSnapshot = 0` therefore relaxes for PR2** to "no
new `onSnapshot` introduced, and the foundation + 1 hook already migrated".
Full removal lands in the Phase 2.5 follow-up PR.

### 3.0 Bridge, not destination

Firestore's `onSnapshot` gave us free push-based realtime. Postgres has no
equivalent out of the box. This ADR replaces it with **SWR polling** — known
boring tech, ships the migration. SWR adds dedup (multiple components reading
the same hook = 1 request) and `revalidateOnFocus` (tab-switch refresh) so the
"feel" stays acceptable.

Polling is the **interim bridge**, not the destination. The proper answer is
writer-driven push (writers emit events on mutation, backend fans out via
SSE). That belongs in a separate ADR — see §11 follow-ups. Building it in
this ADR couples scope (storage swap + new realtime architecture + new
failure modes) and stalls the migration. Defer.

### 3.1 Lists → SWR polling

Each hook moves from `onSnapshot` to `useSWR(url, fetcher, { refreshInterval: N, revalidateOnFocus: true })`.
URL points at a JSON GET endpoint (existing or new) returning the same shape.

Two tiers based on data churn + user attention:

| Tier | Interval | Hooks |
|---|---|---|
| Live-changing | **1 s** | `use-process-instances`, `use-tasks`, `use-monitoring`, `use-agent-runs` (both collection + single), `use-namespace-role`, cowork session listener (page-level) |
| Retro / static | **5 s** | `use-audit-events`, `use-workflow-definitions`, `use-namespace`, `use-all-user-namespaces` |

Rationale:
- 1 s tier = surfaces user actively watches during a run (active workflow, live
  tasks, monitoring dashboards, agent execution). Sub-second push would be
  better but 1 s + SWR dedup is "feels live" without the push infra.
- 5 s tier = retrospective (`audit`), near-static (`workflow-definitions` edited
  on deploy cycles; `namespace` changes are admin actions), or rarely-changing
  (`all-user-namespaces` = membership churn).
- No `revalidateOnFocus: false` overrides — every hook benefits from
  tab-return refresh.

`use-collection.ts` (Firestore-shaped base wrapping `onSnapshot` with
`QueryConstraint[]`) is replaced per-consumer: each hook becomes its own
explicit `useSWR(url, fetcher, { refreshInterval })` call. `use-collection.ts`
deletes when all 9+ consumers (including 2 direct callers in
`components/tasks/`) have migrated. The Firestore abstraction does not
translate to REST — a "generic SWR adapter" would be a from-scratch
rebuild + 9 caller rewrites for no migration value.

### 3.2 Page-component `onSnapshot` listeners

Three page components subscribe directly (not via hooks). Move into hooks
first, then convert per §3.1:

- [`packages/platform-ui/src/app/(app)/[handle]/settings/page.tsx`](../../packages/platform-ui/src/app/(app)/[handle]/settings/page.tsx) — member list listener → `useSWR` (5 s, admin surface)
- [`packages/platform-ui/src/app/(app)/[handle]/tasks/[taskId]/page.tsx`](../../packages/platform-ui/src/app/(app)/[handle]/tasks/[taskId]/page.tsx) — single task listener → `useSWR` (1 s, live)
- [`packages/platform-ui/src/app/(app)/[handle]/cowork/[sessionId]/page.tsx`](../../packages/platform-ui/src/app/(app)/[handle]/cowork/[sessionId]/page.tsx) — cowork session doc → `useSWR` (1 s, feeds chat + voice views with turns + artifact)

### 3.3 Live agent run logs

Today: `agent-log-viewer.tsx` fetches from `/api/step-logs` (file serve,
read-once). **No live streaming exists today.** PLAN previously proposed a
new SSE endpoint backed by Redis pub/sub; dropped. Stays fetch-once + SWR
1 s refresh while the run is active (re-fetch on `status` change).

A real live-tail surface lands in the realtime-push ADR (§11) if usage
patterns warrant it.

### 3.4 Cowork chat + voice

Phase 3.1 (PR #525) already migrated cowork to platform-api handlers, deleted
the unused `/api/cowork/:sid/message` SSE route, and moved `/chat` (JSON,
tool-loop) to a handler. Streaming overhaul is tracked in issue #516.

This ADR's cowork work = (a) page-level Firestore session listener → SWR
(see §3.2), (b) Postgres `cowork-session` repo behind the existing
`CoworkSessionRepository` interface (see §5.2 #8). No streaming work here.

Voice (OpenAI Realtime, WebRTC direct from browser) — out of scope. No
platform-side streaming during conversation.

## 5. Repositories — implementation checklist

### 5.1 Interfaces — all extracted

All 16 Firestore repos now ship a corresponding interface in
`packages/platform-core/src/`. Half live under `interfaces/` (legacy
location), half under `repositories/` (newer location); both are re-exported
from `index.ts`. **No extraction work remains** for PR2 — confirmed after
the tracer (PR1) audit.

In-memory doubles in `packages/platform-core/src/testing/` exist for every
repo that has shipped a Postgres impl or is in the queue; missing ones get
added one-per-repo (e.g. `InMemoryNamespaceRepository` shipped with the
workspace repo in this PR).

### 5.2 Build order (Postgres implementations)

`workspaces` lands first — it is the table every other tenant-scoped table
FKs into. Beyond that, roughly small/leaf → large/root, so each step is
independently testable and each new FK can resolve against a peer that
already exists in the schema.

1. `workspace` (+ `workspace-members`) — the FK root for every tenant-scoped
   table below
2. `tool-catalog` (already shipped as the §0 tracer; FK to `workspaces` added
   when that table lands)
3. `audit`
4. `agent-oauth-token` + `oauth-provider` (mutually referencing)
5. `cron-trigger-state`
6. `agent-run`
7. `human-task`

## 6. Schema & code translation crib sheet

| Firestore-ism | Postgres equivalent |
|---|---|
| `FieldValue.serverTimestamp()` | `default now()` on insert; `now()` in `UPDATE … SET updated_at = now()` |
| `FieldValue.arrayUnion(turn)` (cowork) | `INSERT INTO cowork_turns …` (normalized) |
| `Timestamp` coercion to ISO string | `timestamptz` is native — no coercion |
| `batch.commit()` (`writeBatch`) | `BEGIN … COMMIT` transaction |
| `runTransaction(async tx => …)` (secrets) | `BEGIN ISOLATION LEVEL SERIALIZABLE … COMMIT` |
| `where('foo', 'in', [arr])` capped at 30 | Native `WHERE foo = ANY($1)` — drop the chunking loop |
| `collectionGroup('members').where('uid', …)` (with fallback when index missing) | `SELECT … FROM workspace_members WHERE uid = $1` — drop the fallback |
| Firestore composite indexes (`firestore.indexes.json`) | Equivalent btree indexes on the new tables (sketched in §1.2) |

## 8. Cutover (big-bang)

### 8.1 Pre-cutover

1. All Postgres-backed repositories shipped behind a `STORAGE_BACKEND=firestore|postgres`
   feature flag. Default `firestore` during the implementation phase.
2. CI runs the L2 parity suite + L3 E2E suite against **both** backends. Green
   on both is a merge prerequisite for each repo migration PR.
3. Postgres production instance provisioned in the target environment; backup
   tooling configured.

### 8.2 Cutover window (planned downtime)

1. Announce maintenance window (low-traffic time).
2. Mark the app read-only via a feature flag.
3. Snapshot Firestore (export to GCS).
4. Run the migration script (`scripts/migrate-firestore-to-postgres/main.py`):
   - Streams the Firestore export.
   - Maps each document to the target Postgres row.
   - `INSERT … ON CONFLICT DO NOTHING` for idempotency.
   - Updates per-table counts + writes them to `migration_log` for audit.
5. Run the verification step (`scripts/migrate-firestore-to-postgres/verify.py`):
   - Row counts match per table.
   - 50 random rows per table sampled and compared field-by-field.
6. Flip `STORAGE_BACKEND=postgres` in env. Restart application + workers.
7. Smoke test: log in, list workflows, run one workflow end-to-end, open one
   cowork session, send one human task, see one audit row.
8. Clear the read-only flag. End maintenance window.

### 8.3 Rollback

If verification or smoke fails:

1. Set `STORAGE_BACKEND=firestore` again, restart.
2. Restore Firestore from the snapshot if any writes leaked to Postgres after
   the cut.
3. Triage with full `migration_log` + sample diffs. Retry next window.

### 8.4 Post-cutover (within 2 weeks)

- Delete `packages/platform-infra/src/firestore/`.
- Remove the `STORAGE_BACKEND` flag (always Postgres).
- Drop `firestore.rules`, `firestore.indexes.json`, and the Firestore stanza
  of `firebase.json` (keep `firebase.json` for the still-active Firebase Auth
  flow until ADR-0002 lands).
- Remove all `NEXT_PUBLIC_FIREBASE_*` and `FIRESTORE_EMULATOR_HOST` env vars
  associated with data, keeping only those required by Firebase Auth.
- Update `README.md`, `.env.example`, and `CHANGELOG.md`.

## 8.5 Ongoing migrations

The cutover script in §8.4 is a one-shot **data** move: Firestore →
Postgres, run once during the maintenance window. After cutover, schema
changes ship via drizzle-kit migrations on the normal PR / deploy path.

**Authoring a migration.** A developer edits the Drizzle schema in
`packages/platform-infra/src/postgres/schema/` and runs
`pnpm --filter @mediforce/platform-infra db:generate`. drizzle-kit diffs
the schema against the existing migrations folder, emits a new
`NNNN_description.sql`, and appends to `migrations/meta/_journal.json`.
Both files are committed together. The journal is the source of truth for
"what should be applied"; see the rename rule in §10.1 for branch
collisions.

**Applying migrations.** Migrations live *outside* the Next.js process —
they run as a separate Node script in production and as a developer
command locally. Both paths converge on Drizzle's `migrate()` which reads
the journal, runs each pending file inside a transaction, and records it
in `drizzle.__drizzle_migrations` (schema `drizzle`, table
`__drizzle_migrations` — two underscores, drizzle's own convention).
Running twice is a no-op for already-applied migrations.

- **Production (Docker):**
  [`packages/platform-ui/Dockerfile`](../../packages/platform-ui/Dockerfile)
  wraps the container `CMD`:
  ```sh
  node packages/platform-infra/scripts/migrate.mjs && node packages/platform-ui/server.js
  ```
  The script (
  [`migrate.mjs`](../../packages/platform-infra/scripts/migrate.mjs)
  ) exits 0 when `STORAGE_BACKEND != postgres`, fails the container start
  on any migration error (fail-stop — half-migrated schema is worse than
  "container failed to start"), and otherwise applies pending migrations
  via drizzle's runtime migrator.

- **Local dev:** `pnpm dev:postgres` runs `pnpm db:migrate` before
  booting the dev server (one-command setup). For custom env or
  debugging, `pnpm db:migrate` directly works the same way. Both
  forward to drizzle-kit CLI inside `packages/platform-infra`. See
  [`docs/postgres-local-dev.md`](../postgres-local-dev.md).

- **Operator path (pharma on-prem):** same `pnpm db:migrate` invocation
  if customers want to gate migrations through their change-management
  process. Or, run the same `migrate.mjs` script inside their
  Docker image at any point — the ledger guarantees idempotency.

**Why the script-before-server pattern instead of in-process auto-migrate.**
An earlier attempt routed migrations through Next.js's
`instrumentation.register()` boot hook, but Turbopack's instrumentation
pipeline does not honour `transpilePackages` for workspace imports.
That forced `@ts-expect-error` escapes on relative `.ts` imports and
duplicated `postgres` / `drizzle-orm` as `platform-ui` direct deps. The
standalone-script approach drops both workarounds and matches the
canonical Rails/Django/Prisma pattern (migrate as a separate process
step, not a side-effect of app boot).

**Multi-replica future.** When we scale `platform-ui` past one replica,
the same script becomes a one-shot init-container in compose
(`depends_on: { migrate: { condition: service_completed_successfully } }`)
so exactly one process ever runs migrations per deploy. Tracked as a
follow-up; deferred until we actually run multi-replica.

## 10.1. Migration filename rule

Drizzle-kit names migrations `NNNN_description.sql` and tracks them in
`migrations/meta/_journal.json`. Two PRs branching from main both pick the
next free `NNNN` — the second to merge has to rebase its migration onto
`NNNN+1` (rename the SQL file and re-run `pnpm db:generate` so the journal
matches). CI does not detect the collision today; a duplicate `idx` in
`_journal.json` is the canary on rebase.
